# Veritas Claude Learnings

## 2026-03-02 — Red Team Audit: Unified Profile Architecture

### Root Causes Found

1. **RLS "Service role" policy is a misnomer.** Supabase service role bypasses RLS completely. A policy with `using(true)` applies to ALL authenticated users. Remove open policies; let service role bypass implicitly.

2. **`listUsers()` without email filter breaks at scale.** Defaults to page 1/50 rows. At 1000+ users it misses matches, creating duplicate accounts. Use `getUserByEmail()` instead.

3. **Server Actions accepting client-supplied IDs must verify ownership.** `supabaseAdmin` ignores RLS. Pattern: `getCallerUserId()` → verify `row.user_id === callerId`.

4. **Debug `/api/*` routes must be gated.** `fix-profile` was public with service role. Gate with `x-admin-secret` header vs `ADMIN_SECRET` env var.

5. **Demo fallback in production = data leak.** Never render real data as auth fallback. Use `redirect()`.

6. **`checkIsCreator()` must check auth session first, then viewer cookie.** Mismatched resolution causes ghost states after role upgrades.

7. **`revalidatePath('/', 'layout')` required after role upgrades** to flush Next.js server component cache.

8. **`veritas_user` cookie needs `secure: true` + `sameSite: 'lax'`** in production.

9. **Token verification: description meta tag primary, rawHtml fallback required.** YouTube does NOT reliably populate `<meta name="description">` with the user-edited channel description — it can be truncated or omitted. rawHtml fallback is safe because the token format (`VERITAS-XXXXXXXX`) is already bound to a specific `email + channel_url` pair in `verification_requests`, preventing cross-channel false positives.

### Stripe Collision Guidance (Pre-emptive)
- `stripe_customer_id` (billing) → `profiles` table
- `stripe_connect_account_id` (payouts) → `creators` table
- Creator claim INSERT must never touch `stripe_customer_id`
